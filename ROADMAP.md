## Phase 4 — WebSocket Real-Time (2-3 недели)

### Цель
Перевести скринер из режима "ручной скан каждые 60с" в режим "непрерывный мониторинг через WebSocket с обновлением каждые 5-10 секунд".

### Архитектура
```
                    ┌─────────────────┐
                    │  MEXC WebSocket  │
                    │  depth + deals   │
                    └────────┬────────┘
                             │
                    ┌────────▼────────┐
                    │  ws_monitor.py   │
                    │  (background     │
                    │   thread)        │
                    └────────┬────────┘
                             │ записывает в
                    ┌────────▼────────┐
                    │  SQLite (storage)│
                    └────────┬────────┘
                             │ читает при rerun
                    ┌────────▼────────┐
                    │  Streamlit UI    │
                    │  (autorefresh    │
                    │   каждые 5-10с)  │
                    └─────────────────┘
```

### Задачи
1. **Запуск ws_monitor в background thread** при старте Streamlit
2. ws_monitor записывает в ScannerDB: стакан, сделки, mover events
3. Streamlit читает из ScannerDB при каждом autorefresh
4. DensityTracker загружается из БД при старте (не с нуля)
5. Screener пересчитывается автоматически каждые 30с

### Технические детали
- `threading.Thread(target=run_ws_monitor, daemon=True)` в app.py
- ws_monitor подписывается на топ-20 монет из последнего скана
- Смена списка подписок при каждом полном скане
- Таймаут WS: 30 секунд → переподключение

### Результат
- Recovery time заполняется за 10-15 минут вместо часов
- Drift detection на реальных данных каждые 5 секунд
- Поведение стенок (retreater/aggressor) определяется за 2-3 минуты

## Phase 5 — Advanced Analytics (1-2 месяца)

### 5.1 Machine Learning скоринг
- Когда накопится 30+ дней данных в SQLite
- Обучение XGBoost на: все текущие метрики → "был_ли_прибыльный_цикл"
- Замена эвристического скоринга на data-driven модель

### 5.2 Спектральный анализ таймингов
- FFT (Fast Fourier Transform) на интервалы между сделками
- Обнаружение скрытых периодичностей: 500мс, 1с, 5с, 30с тактов
- Детекция clock-driven роботов vs event-driven

### 5.3 Карта ликвидности (Liquidity Heatmap)
- Визуализация: время × цена × объём стенок
- Показывает как стенки двигаются во времени
- Мгновенно видно: retreater, aggressor, passive MM

### 5.4 Кросс-корреляция между монетами
- Если один ёрш отключился и через 5 минут появился на другой монете → один и тот же оператор
- Кластеризация роботов по поведению → "портфель" оператора

## Phase 6 — Trading Assistant (2-3 месяца)

### 6.1 Полуавтоматический подсказчик
- Видит монету с активным Hunter Pattern
- Показывает: "Рекомендую: продай X по Y, целевая покупка при Z"
- Рекомендуемый объём: 25% от BID-стенки
- Отслеживает исполнение

### 6.2 Telegram/Discord алерты
- Новая S-монета → уведомление
- Изменение grade (B→A, A→F) → уведомление
- Hunter cycle detected → уведомление
- Bot offline → уведомление

### 6.3 Backtesting
- Симуляция squeeze-стратегии на исторических данных из SQLite
- Валидация скоринга: корреляция score ↔ реальная прибыль
- Автоматическая настройка весов блоков скоринга

## Phase 7 — Масштабирование (3-6 месяцев)

### 7.1 Multi-exchange
- Расширение на другие биржи: Gate.io, BitMart, LBANK
- Единый формат данных, exchange-agnostic анализ
- Сравнение одной и той же монеты на разных биржах

### 7.2 React/Vue frontend
- Полная замена Streamlit на кастомный frontend
- WebSocket push вместо polling
- Задержка < 100мс
- Мобильная версия

### 7.3 API для внешних инструментов
- REST API для получения результатов скринера
- Интеграция с TradingView (PineScript alerts)
- Webhook → торговые терминалы

**Возможен ли автоскан в реальном времени?**
Да. Phase 4 (ws_monitor в background thread + autorefresh) даёт задержку 5-10 секунд.
Для стратегии "поджать ёрша" это более чем достаточно — цикл ёрша длится минуты.
Полная реализация Phase 4 — 1-2 недели разработки.
